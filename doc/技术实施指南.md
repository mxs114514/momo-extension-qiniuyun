# AI 同声传译助手技术实施指南

> 本文记录架构、模块边界、音频处理、状态管理、测试、安全和阶段迁移等具体
> 实施指导。技术选型摘要见 `技术栈.md`。

## 1. 技术选型原则

技术栈围绕 `prd.md` 的阶段目标逐步演进，不在一期提前引入历史记录、复杂状态
管理、AI 角色或 Live2D 等非必要能力。

选型遵循：

1. 翻译能力优先，先验证完整实时链路。
2. 观看体验优先，音频处理和字幕更新不能明显影响页面播放。
3. 核心能力与运行载体解耦，方便从普通网页迁移到浏览器插件。
4. 音频来源可替换，一期使用麦克风，后续增加浏览器标签页音频。
5. 流式结果按句子覆盖更新，避免字幕重复。
6. 测试覆盖与风险匹配，云服务真实联调不进入自动化测试。
7. 长期密钥不得进入正式发布产物。

## 2. 产品阶段与技术路线

| 阶段 | 产品目标 | 技术重点 |
| --- | --- | --- |
| 一期原型 | 验证实时英译中完整链路 | Vite 网页、麦克风、腾讯云实时语音翻译 |
| 一期正式形态 | 浏览器插件中观看视频并显示字幕 | WXT、Manifest V3、标签页音频、字幕覆盖层 |
| 二期 | 历史记录与导出 | IndexedDB、Dexie、Blob API |
| 三期 | AI 翻译官模式 | 展示模式切换、角色资源、可选 LLM |
| 四期 | Live2D 翻译官 | Live2D SDK、性能隔离、降级策略 |

当前优先实施“一期原型”。浏览器插件正式形态是下一阶段迁移目标，不应阻塞
腾讯云实时语音翻译的首次接入。

## 3. 一期原型技术栈

### 3.1 核心技术

| 层级 | 选型 | 用途 |
| --- | --- | --- |
| 前端框架 | React 19 | 页面和字幕 UI |
| 开发语言 | TypeScript 6 | 领域模型、协议和浏览器 API 类型约束 |
| 构建工具 | Vite 8 | 本地开发和构建 |
| 包管理 | pnpm | 依赖与脚本管理 |
| 样式 | Tailwind CSS 4 + 少量 CSS | 页面布局、状态和字幕样式 |
| 音频采集 | MediaDevices | 获取麦克风音频 |
| 音频处理 | Web Audio API + AudioWorklet | 降采样、PCM 转换和分包 |
| 实时传输 | 原生 WebSocket | 连接腾讯云实时语音翻译 |
| 签名 | Web Crypto API | HMAC-SHA1 签名 |
| 云服务 | 腾讯云实时语音翻译 | 同时返回英语原文和中文译文 |
| 单元测试 | Vitest | 纯函数、状态机和协议测试 |
| 组件测试 | Testing Library | React 行为和可访问性测试 |
| 端到端测试 | Playwright | 页面基本流程测试 |

一期不强制使用 Zustand、Zod、shadcn/ui、IndexedDB 或 WXT。

### 3.2 一期运行形态

```text
Vite React 页面
    │
    ├── 开始、停止和状态展示
    ├── 只展示中文译文
    │
    ▼
SpeechTranslationController
    │
    ├── MicrophoneAudioSource
    │     └── 16kHz / 16bit / 单声道 PCM
    │
    ├── TencentSpeechTranslateClient
    │     └── 腾讯云实时语音翻译 WebSocket
    │
    └── TranscriptStore
          ├── 保存英语原文
          └── 保存并展示中文译文
```

### 3.3 模块职责

#### `MicrophoneAudioSource`

- 请求麦克风权限。
- 使用 `AudioContext` 和 `AudioWorkletNode` 获取音频样本。
- 将音频转换为腾讯云要求的 16000Hz、16bit、小端、单声道 PCM。
- 约每 200ms 输出一个 6400 字节数据包。
- 停止时释放音轨、音频节点和 `AudioContext`。

#### `TencentSpeechTranslateClient`

- 构造腾讯云连接参数和签名。
- 建立实时语音翻译 WebSocket。
- 发送 PCM 数据。
- 解析 `source_text` 和 `target_text`。
- 停止时发送 `{"type":"end"}` 并等待最终响应。

#### `TranscriptStore`

- 使用 `sentence_id` 标识句子。
- 同一句的新结果覆盖旧结果，不进行字符串重复追加。
- 同时保存英语原文和中文译文。
- 一期 UI 只读取中文译文。

#### `SpeechTranslationController`

- 管理请求权限、连接中、翻译中、停止中和异常状态。
- 协调音频源、翻译客户端和字幕 Store。
- 统一处理错误和资源清理。
- 对 React 暴露稳定的状态快照。

## 4. 音频输入设计

从一期开始定义统一音频源接口：

```typescript
interface AudioSource {
  start(onChunk: (chunk: Int8Array) => void): Promise<void>
  pause(): Promise<void>
  resume(): Promise<void>
  stop(): Promise<void>
}
```

一期实现：

```text
MicrophoneAudioSource
```

浏览器插件阶段增加：

```text
TabAudioSource
```

按需增加：

```text
MixedAudioSource
```

所有音频源统一输出 PCM 数据，使腾讯云客户端不需要了解音频来自麦克风、
浏览器标签页还是混合音轨。

`pause()` 暂停音频处理和数据输出，但保留当前音频设备；`resume()` 恢复同一
音频源。`stop()` 才彻底释放音轨和音频上下文，避免恢复时再次请求权限。

## 5. 腾讯云实时语音翻译

一期直接使用腾讯云实时语音翻译 WebSocket：

```text
wss://asr.cloud.tencent.com/asr/speech_translate/<appid>
```

固定方向和模型：

```typescript
{
  source: 'en',
  target: 'zh',
  voice_format: 1,
  trans_model: 'hunyuan-translation-lite',
}
```

服务返回：

- `source_text`：英语原文。
- `target_text`：中文译文。
- `sentence_id`：句子标识。
- `sentence_end`：当前句是否完成。
- `final`：整个会话是否结束。

一期内部保留两种文字，页面只显示 `target_text`。

不采用“腾讯云 ASR + 文本翻译”串联方案，因为该方案网络请求更多、延迟更高，
并且需要自行处理频繁变化的中间识别结果。

仓库中的 `tencentcloud-speech-sdk-js/asr` 面向普通 ASR，只作为录音、PCM 和签名
逻辑的参考，不作为实时语音翻译客户端直接依赖。

## 6. 自动纠错的技术定义

PRD 一期中的自动纠错实现为“流式结果修订”，不额外调用大模型：

1. 腾讯云持续返回当前句的完整识别和翻译结果。
2. 使用 `sentence_id` 找到当前句。
3. 新的中间结果覆盖该句旧内容。
4. `sentence_end=true` 后将句子标记为最终结果。
5. 已结束句子按顺序保留。

```text
Welcome
  → 欢迎

Welcome everyone
  → 欢迎大家
```

界面应把“欢迎”替换为“欢迎大家”，不能显示为“欢迎欢迎大家”。

三期如果需要术语润色、上下文改写或摘要，再评估 LLM，不纳入一期。

## 7. 一期状态管理

一期采用独立 Controller 和 React 原生订阅能力：

```text
idle
  → requesting-permission
  → connecting
  → translating
  → paused
  → translating
  → stopping
  → idle

任意状态 → error
```

暂停时调用 `AudioSource.pause()` 并停止向腾讯云发送音频，但保留 WebSocket
会话。恢复时调用 `AudioSource.resume()`，继续向同一会话发送数据。若腾讯云
因暂停时间过长而断开连接，则进入异常状态并提示用户重新开始。

React 可使用 `useSyncExternalStore` 订阅 Controller。

一期不引入 Zustand，原因是当前只有单页面和单会话状态。迁移到浏览器扩展后，
如果 Popup、Side Panel、Content Script 需要共享复杂 UI 状态，再评估 Zustand。

浏览器扩展跨上下文状态不能只依靠 Zustand，仍需使用 Chrome 消息通信和持久化
API。

## 8. 一期 UI 选型

一期页面仅包含：

- 开始和停止按钮。
- 翻译状态。
- 中文字幕区域。
- 错误提示。
- 本地密钥安全警告。

使用 React、Tailwind CSS 和少量原生 CSS 即可。

Tailwind CSS 4 通过现有 `@tailwindcss/postcss` 接入，在全局样式入口中添加：

```css
@import 'tailwindcss';
```

布局、间距、颜色、排版、响应式状态和按钮样式优先使用 Tailwind 工具类。
`src/index.css` 只保留 Tailwind 导入、全局变量和基础元素样式；`src/App.css`
只保留工具类不适合表达的少量自定义样式。

shadcn/ui 保留为长期 UI 组件方案，但不作为一期强制依赖。进入设置页、历史
记录页或 Side Panel 后，再按实际组件需求引入 Button、Dialog、Tabs 等组件。

## 9. 一期数据校验

一期不强制引入 Zod。

原因：

- 配置字段数量少，可使用显式 TypeScript 校验函数。
- 腾讯云 WebSocket 响应频率高，只需检查必要字段和错误码。
- 当前没有复杂表单、持久化数据迁移或外部 JSON 导入。

二期出现历史数据版本、导入导出或复杂配置后，再评估 Zod。

## 10. 一期测试策略

### 单元测试

使用 Vitest 测试：

- PCM 降采样和 16bit 转换。
- 6400 字节分包。
- 腾讯云参数排序和签名。
- WebSocket 消息解析。
- 字幕按 `sentence_id` 覆盖。
- 状态机和资源释放。

### 组件测试

使用 Testing Library 测试：

- 状态文案和按钮变化。
- 页面只显示中文，不显示英语原文。
- 中间结果被覆盖。
- 错误提示和重试行为。

### 端到端测试

使用 Playwright 测试页面渲染和交互外壳。

自动化测试不连接真实腾讯云，也不访问真实麦克风。通过假客户端或浏览器适配器
模拟结果。

真实腾讯云账号、麦克风权限和网络质量通过手工联调验收。

## 11. 一期凭证安全

一期允许使用 `.env.local` 配置固定密钥，仅用于本地开发和比赛演示：

```dotenv
VITE_TENCENT_APP_ID=
VITE_TENCENT_SECRET_ID=
VITE_TENCENT_SECRET_KEY=
```

必须遵守：

- 不得向 Git 提交真实密钥。
- 不得在日志、错误信息或 UI 中打印 SecretKey。
- 不得部署或公开分发包含长期密钥的构建产物。
- 使用单独的测试子账号、最小权限和额度告警。
- `.env.local` 只能避免误提交，不能阻止浏览器用户读取密钥。
- 生产构建应主动拒绝前端长期密钥配置。

正式浏览器插件发布前，必须迁移到服务端代理或经过腾讯云实际验证的短期凭证
方案。

## 12. 一期正式浏览器插件形态

完成普通网页原型后，再将运行载体迁移为：

| 层级 | 选型 |
| --- | --- |
| 扩展框架 | WXT |
| 浏览器规范 | Manifest V3 |
| 构建工具 | WXT 内部 Vite |
| 交互入口 | Popup 或 Side Panel |
| 字幕显示 | Content Script |
| 持续音频处理 | Offscreen Document |
| 状态与路由 | Background Service Worker + Chrome Messaging |
| 音频来源 | `chrome.tabCapture` 对应的 `TabAudioSource` |

推荐架构：

```text
Popup / Side Panel
    │ 开始、暂停、停止
    ▼
Background Service Worker
    │ 状态管理和消息路由
    ▼
Offscreen Document
    ├── TabAudioSource
    ├── AudioWorklet
    └── 腾讯云实时语音翻译 WebSocket
    │
    ▼
Content Script
    └── 在视频页面显示中文字幕
```

Manifest V3 的 Service Worker 可能被浏览器终止，因此不把持续音频处理和
WebSocket 完全放在 Service Worker 中。

迁移时应复用：

- `AudioSource` 接口。
- `TencentSpeechTranslateClient`。
- `TranscriptStore`。
- 状态机和错误模型。

迁移时替换：

- `MicrophoneAudioSource` → `TabAudioSource`。
- 普通 React 页面 → Popup、Side Panel 和 Content Script。
- 页面内状态通信 → Chrome Messaging。

## 13. 二期：历史记录与导出

PRD 二期需要记录列表、记录详情和内容导出。

推荐增量技术：

| 能力 | 选型 |
| --- | --- |
| 结构化本地数据库 | IndexedDB |
| 数据访问层 | Dexie |
| 少量设置 | `chrome.storage.local` |
| 导出 | Blob API + `URL.createObjectURL()` |
| 导出格式 | Markdown 为主，TXT 和 JSON 可选 |
| 数据校验 | 按复杂度评估 Zod |

职责划分：

- `chrome.storage.local` 保存语言、字幕样式等少量配置。
- IndexedDB 保存会话和大量字幕句子。
- Dexie 管理查询、索引和版本迁移。
- 导出层从领域数据生成 Markdown、TXT 或 JSON。

不建议将完整会话长期堆积在 `chrome.storage.local`。

## 14. 三期：AI 翻译官模式

三期首先是展示模式升级，不应改变实时翻译核心链路。

推荐增量技术：

- React 展示模式组件。
- CSS 动画或轻量动画库。
- 独立角色资源管理。
- 经典字幕模式与角色对话模式共享同一字幕 Store。
- Zustand 仅在多页面 UI 状态明显复杂时引入。

只有出现以下需求时才接入 LLM：

- 专业术语润色。
- 跨句上下文改写。
- 内容摘要。
- 翻译官角色对话。

LLM 请求不能阻塞实时字幕主链路。失败时必须降级为腾讯云原始中文译文。

## 15. 四期：Live2D 探索

Live2D 必须作为独立可关闭模块接入。

技术要求：

- 按需动态加载 Live2D SDK 和模型资源。
- 与音频采集、WebSocket 和字幕状态解耦。
- 监控 CPU、GPU、内存和页面帧率。
- 低性能设备可关闭动画或回退到静态角色。
- Live2D 异常不能影响实时翻译。

在实时翻译和历史记录稳定前，不进入该阶段。
